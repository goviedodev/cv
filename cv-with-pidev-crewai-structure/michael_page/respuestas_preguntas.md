## Pregunta 4: Comente experiencia con patrones de diseño y arquitectura hexagonal. Explicar

### Patrones de Diseño

Aplico **patrones de diseño (GoF)** y **patrones empresariales (PoEAA)** de forma consistente en todos mis proyectos. No los uso como recetas rígidas sino como herramientas para resolver problemas recurrentes de manera predecible y mantenible.

#### Patrones Creacionales

| Patrón | Dónde lo he aplicado | Explicación breve |
|--------|---------------------|-------------------|
| **Singleton** | Beans de Spring (por defecto), conexiones a BD, manejadores de logging | Garantiza una única instancia de un objeto. Spring lo maneja por defecto con el scope singleton. |
| **Factory Method** | Creación de DTOs desde entidades, fábrica de notificaciones | Define una interfaz para crear objetos, pero deja que las subclases decidan qué clase instanciar. Lo uso en mappers de entidades a DTOs. |
| **Builder** | Construcción de consultas dinámicas, objetos complejos (ej: filtros de búsqueda) | Separa la construcción de un objeto complejo de su representación. Lo aplico en Criteria API y en queries parametrizadas. |

#### Patrones Estructurales

| Patrón | Dónde lo he aplicado | Explicación breve |
|--------|---------------------|-------------------|
| **Adapter** | Wrappers para APIs externas, integración con servicios legacy | Convierte la interfaz de una clase en otra interfaz que el cliente espera. Lo uso para aislar el dominio de librerías externas. |
| **Proxy** | Spring AOP (@Transactional, @Cacheable), seguridad con @PreAuthorize | Proporciona un sustituto o representante de otro objeto para controlar el acceso a él. Spring lo usa extensivamente en sus proxies AOP. |
| **Facade** | Servicios de aplicación (Service Layer), controladores REST | Proporciona una interfaz simplificada para un conjunto de interfaces más complejas. Mis @Service actúan como fachada entre controllers y repositorios. |
| **Decorator** | Capas de caching, logging transversal, manejo de excepciones | Añade responsabilidades adicionales a un objeto dinámicamente. Spring lo implementa mediante AOP para @Transactional, @Cacheable. |

#### Patrones de Comportamiento

| Patrón | Dónde lo he aplicado | Explicación breve |
|--------|---------------------|-------------------|
| **Strategy** | Algoritmos de validación, cálculos de precios, generación de reportes | Define una familia de algoritmos intercambiables. Lo uso en la lógica de cálculo de pagos y generación de reportes en diferentes formatos. |
| **Template Method** | Spring (JdbcTemplate, RestTemplate), procesos batch con Spring Batch | Define el esqueleto de un algoritmo en una operación, delegando pasos a subclases. Spring lo usa extensivamente en sus *Template classes. |
| **Observer** | Eventos de dominio con ApplicationEventPublisher (Spring Events) | Define una dependencia uno-a-muchos para que cuando un objeto cambie de estado, todos sus dependientes sean notificados. Lo uso para desacoplar flujos: ej, cuando se crea un pago, notificar a contabilidad y al cliente. |
| **Chain of Responsibility** | Filtros de Spring Security, middlewares, pipelines de validación | Permite pasar una solicitud a través de una cadena de handlers. Spring Security lo implementa con su FilterChain. |

#### Patrones Empresariales (PoEAA — Martin Fowler)

| Patrón | Aplicación |
|--------|-----------|
| **Repository** | Spring Data JPA (@Repository) — abstrae el almacenamiento de datos |
| **Service Layer** | @Service — encapsula la lógica de negocio |
| **DTO (Data Transfer Object)** | Separación entre entidades JPA y objetos que expone la API |
| **Unit of Work** | Gestionado por JPA/Hibernate (EntityManager) — rastrea cambios y los sincroniza |
| **Lazy Loading** | FetchType.LAZY en JPA — carga diferida de relaciones |
| **DAO (Data Access Object)** | Spring Data JPA Repository es una implementación moderna de DAO |

### Arquitectura Hexagonal (Ports & Adapters)

#### ¿Qué es?

La **Arquitectura Hexagonal** (también llamada **Ports & Adapters**), propuesta por **Alistair Cockburn**, es un patrón arquitectónico cuyo objetivo principal es **aislar el núcleo de la aplicación (dominio) de los detalles técnicos externos** (bases de datos, APIs, UI, mensajería).

El nombre "hexagonal" es conceptual — no significa que tenga 6 lados, sino que representa un **núcleo central rodeado de puertos**, donde cada puerto puede tener múltiples adaptadores.

#### Estructura

```
┌──────────────────────────────────────────────────┐
│                   INFRASTRUCTURA                  │
│  (Base de datos, API externa, Kafka, UI, etc.)   │
│                                                   │
│  ┌──────────────────────────────────────────────┐ │
│  │              ADAPTADORES (INPUT)              │ │
│  │  (Controllers REST, handlers CLI, listeners) │ │
│  └──────────────┬───────────────────────────────┘ │
│                 │                                  │
│         ┌───────▼────────┐                         │
│         │    PUERTOS      │                        │
│         │  (Interfaces)   │                        │
│         └───────▲────────┘                         │
│                 │                                  │
│  ┌──────────────▼───────────────────────────────┐  │
│  │              DOMINIO / APLICACIÓN             │  │
│  │   (Lógica de negocio, casos de uso, entidades)  │
│  └──────────────┬───────────────────────────────┘  │
│                 │                                   │
│         ┌───────▼────────┐                          │
│         │    PUERTOS      │                         │
│         │  (Interfaces)   │                         │
│         └───────▲────────┘                          │
│                 │                                   │
│  ┌──────────────▼───────────────────────────────┐  │
│  │             ADAPTADORES (OUTPUT)              │  │
│  │  (Repositorios JPA, clientes HTTP, producers) │ │
│  └───────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────┘
```

#### Componentes Clave

1. **Núcleo / Dominio (Core):** Contiene las entidades de negocio, value objects, y casos de uso. **NO depende de nada externo.** No importa si usas Oracle, PostgreSQL o MongoDB — el dominio no lo sabe.

2. **Puertos (Ports):** Son **interfaces Java** definidas en el dominio. Representan los puntos de entrada (input ports: casos de uso) y salida (output ports: repositorios).

3. **Adaptadores (Adapters):** Son implementaciones concretas de los puertos. Un puerto de repositorio puede tener un adaptador JPA, otro Redis, otro mock para testing.

#### Principio Fundamental: La Regla de Dependencia

Las dependencias **SIEMPRE apuntan hacia adentro** (del adaptador hacia el puerto, del puerto hacia el dominio).

- ✅ **Dominio** → no importa nada externo
- ✅ **Puertos** → son interfaces en el dominio
- ✅ **Adaptadores** → importan y dependen del dominio
- ❌ **Dominio** → nunca debe importar un repositorio JPA o un controller REST

#### Mi Experiencia Práctica con Arquitectura Hexagonal

**Proyecto: Startup de Movilidad (2024 - Presente)**

Implementé Arquitectura Hexagonal completa para una plataforma de movilidad con Java 21 y Spring Boot:

**Estructura de paquetes:**
```
com.mobility.platform/
├── domain/
│   ├── model/          (Entidades: User, Trip, Payment, Vehicle)
│   ├── port/
│   │   ├── input/      (Casos de uso: CreateTripUseCase, ProcessPaymentUseCase)
│   │   └── output/     (Repositorios: TripRepository, PaymentRepository)
│   └── service/        (Implementaciones de casos de uso)
├── adapter/
│   ├── input/
│   │   └── rest/       (Controllers REST)
│   └── output/
│       ├── persistence/ (Repositorios JPA con Spring Data)
│       └── messaging/   (Productores Kafka)
└── shared/
    └── config/          (Configuración general)
```

**Beneficios que obtuve en la práctica:**

1. **Testeabilidad absoluta:** Podía testear los casos de uso sin levantar base de datos, solo con mocks de los puertos de salida. Ejecutaba pruebas unitarias de lógica de negocio en milisegundos.

2. **Cambio de base de datos sin impacto:** Inicialmente usé PostgreSQL, luego migré parte a MongoDB para datos de geolocalización. Solo cambié el adaptador de persistencia — el dominio no se modificó.

3. **Aislamiento del dominio:** La lógica de negocio (cálculo de tarifas, validación de viajes, procesamiento de pagos) está completamente aislada de frameworks y librerías. Si mañana quiero cambiar de Spring Boot a Quarkus, solo cambio los adaptadores.

4. **Claridad arquitectónica:** Cada desarrollador nuevo entendía rápidamente la estructura: "el dominio tiene las reglas de negocio, los adaptadores son solo implementaciones técnicas".

**Proyecto: Seven IT — Sistema de Salud (2017-2020)**

Aunque no lo llamaba "Arquitectura Hexagonal" en ese momento, en la práctica la aplicaba:
- Separación clara entre lógica de negocio (@Service) y acceso a datos (@Repository)
- Interfaces para repositorios, permitiendo cambiar de PostgreSQL a MySQL sin afectar el negocio
- Controladores REST delgados que solo orquestaban, sin lógica de negocio

#### Diferencias con Arquitectura en Capas Tradicional

| Aspecto | Capas Tradicional (Layered) | Hexagonal (Ports & Adapters) |
|---------|---------------------------|------------------------------|
| **Dirección de dependencias** | Solo hacia abajo (Controller → Service → Repository) | Siempre hacia el centro (dominio) |
| **Acoplamiento a BD** | Service depende de Repository (que es JPA) | Service depende de interfaz (puerto), no de JPA |
| **Testeabilidad** | Media — requiere contexto de Spring | Alta — los casos de uso se prueban con mocks simples |
| **Cambio de tecnología** | Dificultad media — hay que reescribir capas | Facilidad alta — solo cambiar adaptadores |
| **Complejidad inicial** | Baja | Media (más archivos, más interfaces) |
| **Mantenibilidad a largo plazo** | Media — el dominio se contamina con anotaciones de framework | Alta — dominio puro, limpio de dependencias externas |

#### Conclusión

La Arquitectura Hexagonal no es una moda, es una **decisión arquitectónica que protege la inversión más valiosa del software: la lógica de negocio**. En mi experiencia:

- En **proyectos pequeños** (menos de 6 meses), a veces no vale la pena el overhead inicial
- En **proyectos medianos a grandes** (1+ años), es una de las mejores decisiones arquitectónicas que se pueden tomar
- **Siempre aplico el principio subyacente:** el dominio no debe depender de frameworks. Aunque no implemente la estructura hexagonal completa, separo siempre las interfaces de las implementaciones.
